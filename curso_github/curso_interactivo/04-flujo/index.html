El flujo de trabajo básico de Git

El ciclo de 4 pasos que vas a repetir miles de veces: editar, agregar, confirmar y subir. Memorízalo: edit → add → commit → push.

Cada vez que modifiques tu proyecto, repetirás este ciclo. Es el flujo de Git. Una vez que lo interiorices, el resto del curso encaja en su lugar.

Analogía del supermercado

Cómo entenderlo de una vez

Imagina que vas a comprar. Los productos en tu carrito son los archivos en staging. Cuando pasas por caja y pagas, ese es el commit. Llevarlo a casa es el push.

Cómo se ve en la terminal

# Ver qué cambió
git status

# Agregar archivos al staging (todos los modificados)
git add .

# Crear un commit con un mensaje descriptivo
git commit -m "agrega validación al formulario de login"

# Subir los cambios al repositorio remoto
git push

Explicación palabra por palabra

Comando¿Qué hace en realidad?
git statusTe enseña una "foto" del estado actual: qué archivos cambiaste, cuáles están listos para commit, si hay algo por sincronizar.
git add .Mete en el "carrito" (staging) todos los archivos modificados. El . significa "todo lo de esta carpeta".
git commit -m "mensaje"Saca una foto de lo que hay en el carrito y la guarda en el historial. El -m es para escribir el mensaje en la misma línea.
git pushLleva tus commits desde tu computadora hasta GitHub.
git pullLo contrario de push: baja los cambios que haya en GitHub a tu PC.

Tip de oro

Ejecuta git status antes y después de cada paso. Es tu brújula: te dice qué archivos están modificados, cuáles están en staging y si hay algo por sincronizar con el remoto.

Flujo extendido cuando trabajas en equipo

git pull                          # 1. Bajas lo último de GitHub
git checkout -b mi-rama           # 2. Creas tu rama de trabajo
# (editas archivos)
git status                        # 3. Ves qué cambió
git add .                         # 4. Preparas todo
git commit -m "qué hiciste"       # 5. Guardas la foto
git push -u origin mi-rama        # 6. Subes tu rama
# (en GitHub) abres un Pull Request
git checkout main                 # 7. Vuelves a la principal
git pull                          # 8. Actualizas
git branch -d mi-rama             # 9. Borras la rama vieja

No te asustes con todos esos pasos: los iremos viendo uno por uno. Lo importante es saber que todo gira alrededor del ciclo de 4 pasos.
