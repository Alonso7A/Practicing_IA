Los estados de un archivo en Git

Untracked, modified, staged, committed, pushed o ignored. Saber identificar cada estado te ahorra horas de confusión.

Cada archivo de tu proyecto pasa por estos estados. Cuando entiendes el ciclo, los mensajes de git status dejan de ser un misterio.

stateDiagram-v2
    direction LR
    [*] --> Untracked: archivo nuevo
    Untracked --> Staged: git add
    Staged --> Committed: git commit
    Committed --> Modified: editar archivo
    Modified --> Staged: git add
    Committed --> Pushed: git push
    Pushed --> [*]
    Modified --> Committed: git restore (descartar)
Fig. 1. Ciclo de estados de un archivo: del nacimiento (untracked) a la nube (pushed).

¿Por qué hay un estado "staged" entre modified y committed?

El truco del staging

El staging area existe para que puedas elegir exactamente qué archivos entran en cada commit. Si modificaste 10 archivos pero solo 3 corresponden a un cambio lógico, puedes hacer git add sólo de esos 3, commitear, y luego repetir con los siguientes. Eso hace que tu historial sea limpio y entendible.

Comandos para moverse entre estados

# De untracked / modified  →  staged
git add archivo.txt
git add .                        # todos los modificados

# De staged  →  modified (quitar del staging sin perder cambios)
git restore --staged archivo.txt

# De modified  →  committed (descartando cambios)
git restore archivo.txt          # ⚠️ pierdes los cambios

# De staged  →  committed
git commit -m "mensaje"

# De committed  →  pushed
git push

# Marcar como ignored (no se hace con comando: edita .gitignore)

Cuidado con git restore

git restore archivo borra los cambios no commiteados. No hay papelera de reciclaje. Antes de usarlo, asegúrate de no necesitar lo que vas a descartar.
