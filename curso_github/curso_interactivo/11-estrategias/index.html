Estrategias de branching: GitHub Flow vs Git Flow

Dos formas distintas de organizar las ramas de un proyecto. Una es simple y moderna; la otra, formal y para releases grandes. Aprende cuál elegir.

Existen varias formas de organizar las ramas en un proyecto. Las dos más conocidas son GitHub Flow (simple) y Git Flow (complejo). Aprende la primera, conoce la segunda.

GitHub Flow (recomendado para empezar)

Una rama principal main siempre desplegable. Cualquier cambio nace en una rama feature, pasa por PR, y se fusiona a main. Simple, rápido, ideal para proyectos personales y CI/CD.

gitGraph
    commit
    commit
    branch feature-a
    commit
    commit
    checkout main
    merge feature-a
    branch feature-b
    commit
    checkout main
    merge feature-b
    commit
Fig. 1. GitHub Flow: una sola rama principal, ramas de feature efímeras.

Git Flow (proyectos grandes con releases)

Múltiples ramas con roles específicos: main (producción), develop (desarrollo), feature/, release/, hotfix/. Más estructurado pero más complejo. Hoy se considera "viejo" y se prefiere GitHub Flow + trunk-based development en la mayoría de equipos modernos.

gitGraph
    commit id: "v1.0"
    branch develop
    commit
    branch feature/login
    commit
    commit
    checkout develop
    merge feature/login
    branch release/1.1
    commit id: "RC"
    checkout main
    merge release/1.1 tag: "v1.1"
    checkout develop
    merge release/1.1
Fig. 2. Git Flow: estructura formal con ramas de release y hotfix.

¿Cuál usar?

Si trabajas en un proyecto personal, una página web o un equipo pequeño: GitHub Flow. Si trabajas en software con versiones numeradas formales (ej: v2.4.1) y necesitas mantener varias releases en paralelo: Git Flow.
