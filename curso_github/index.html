Curso de GitHub: desde la web hasta VS Code

Una guía completa, con teoría, comandos, diagramas y tips, para dominar el control de versiones tanto desde la interfaz web de GitHub como desde la terminal de Visual Studio Code.

¿Qué es Git y qué es GitHub?

Aunque suelen mencionarse juntos, Git y GitHub no son lo mismo. Entender la diferencia es el primer paso para no confundir conceptos.

Definición

Git es un sistema de control de versiones distribuido: un programa que se instala en tu computadora y registra cada cambio que haces en los archivos de un proyecto, permitiéndote volver atrás, comparar versiones y trabajar en paralelo con otras personas.

GitHub es una plataforma en la nube que aloja repositorios Git, añade colaboración (issues, pull requests, code review), respaldo, hosting de páginas, automatizaciones (Actions) y se ha convertido en el portafolio profesional estándar para desarrolladores.

Analogía

Git es el cuaderno donde anotas todos los cambios, quién los hizo y cuándo.

GitHub es la biblioteca en internet donde guardas ese cuaderno, con mesas para reunirse, ventanas para que otros vean tu trabajo y estantes para organizar los capítulos.

Conceptos clave que debes conocer

Antes de empezar a teclear comandos, familiarízate con este vocabulario. Lo encontrarás en cada tutorial, error y conversación sobre Git.

Repositorio (repo)

La carpeta de tu proyecto, junto con todo el historial de cambios. Puede ser local (en tu PC) o remoto (en GitHub).

Commit

Una "fotografía" de tus archivos en un momento dado. Cada commit tiene un mensaje, un autor y un identificador único (hash).

Branch (rama)

Una línea paralela de desarrollo. Te permite trabajar en una funcionalidad sin afectar el código principal.

Merge

Fusionar los cambios de una rama dentro de otra. Es cómo se incorpora tu trabajo a la rama principal.

Push / Pull

push sube tus commits al servidor remoto. pull descarga lo que otros subieron.

Clone

Descargar un repositorio remoto completo a tu máquina para empezar a trabajar en él.

Staging area

Zona intermedia donde colocas los archivos que quieres incluir en tu próximo commit. Se llena con git add.

Pull Request (PR)

Solicitud para fusionar una rama en otra. Es donde ocurre la revisión de código entre colaboradores.

Instalación y configuración inicial

Para trabajar con GitHub necesitas tres cosas: una cuenta en la web, Git instalado en tu PC y un editor de código (idealmente Visual Studio Code).

Paso 1 — Crear cuenta en GitHub

Ve a github.com y pulsa Sign up. Vas a necesitar correo, usuario y contraseña. Confirma tu correo desde el enlace que te envíen.

Paso 2 — Instalar Visual Studio Code

Descárgalo desde code.visualstudio.com. Es el editor más usado y tiene integración nativa con Git, terminal incorporada y miles de extensiones.

Paso 3 — Instalar Git

Descárgalo desde git-scm.com/downloads. En Windows el instalador te hace varias preguntas — la siguiente tabla resume qué elegir si eres principiante:

Opciones recomendadas en la instalación de Git para Windows

Nombre de rama inicial Let Git decide Usará master. Es válido. Hoy se prefiere main pero no es obligatorio.
Ajuste del PATH Git from the command line and 3rd-party software Permite usar Git desde CMD, PowerShell y la terminal de VS Code.

Paso 4 — Configurar tu identidad en Git

Abre la terminal en VS Code (Ctrl+`) y ejecuta:

git config --global user.name "Tu Nombre"
git config --global user.email "tucorreo@ejemplo.com"
git config --list

Importante

El correo que pongas en user.email debería ser el mismo de tu cuenta de GitHub. Si no, los commits aparecerán como "no verificados" o atribuidos a una cuenta fantasma.

Cómo se ve en la terminal

# Ver qué cambió
git status

# Agregar archivos al staging (todos los modificados)
git add .

# Crear un commit con un mensaje descriptivo
git commit -m "agrega validación al formulario de login"

# Subir los cambios al repositorio remoto
git push

Tip de oro

Ejecuta git status antes y después de cada paso. Es tu brújula: te dice qué archivos están modificados, cuáles están en staging y si hay algo por sincronizar con el remoto.

Crear un repositorio desde la web de GitHub

Hay dos caminos: crear el repo en GitHub primero y luego clonarlo, o crearlo localmente y luego conectarlo. Empezaremos por el camino más limpio: crearlo en GitHub primero.

Pasos en la interfaz web

  1. PASO 1: Inicia sesión en github.com. En la esquina superior izquierda pulsa el icono de las tres rayas y selecciona Repositories.
  2. PASO 2: Pulsa New repository. Escribe un nombre (sin espacios, usa guiones) y una descripción breve. Déjalo público al inicio: facilita compartir y aprender, y GitHub Pages necesita repos públicos para hosting gratuito.
  3. PASO 3: Marca Add a README file para que el repo no esté vacío. Opcionalmente agrega un .gitignore según el lenguaje.
  4. PASO 4: Pulsa Create repository. GitHub te mostrará la URL del repo: cópiala.

Conectar tu carpeta local al repo recién creado

Abre tu carpeta en VS Code, abre la terminal y ejecuta:

# Inicializar Git en la carpeta
git init

# Conectar al repo remoto
git remote add origin https://github.com/tu-usuario/tu-repo.git

# Bajar el README/gitignore que se creó en GitHub
git pull origin main

# Crear archivos, agregar, commit y push
git add .
git commit -m "primer commit local"
git push --set-upstream origin main

Cuando aparece error en el primer push

Si te sale error: failed to push some refs, suele ser porque el remoto tiene commits que tú no tienes localmente (el README que creaste). Solución: git pull origin main --allow-unrelated-histories y vuelve a hacer git push.

Crear una carpeta nueva dentro del repo desde la web

Sí, en la interfaz web puedes crear carpetas, pero hay un truco: cuando pulsas Add file → Create new file, en el campo del nombre escribe:

mi-carpeta/archivo.md

El / es lo que convierte mi-carpeta en carpeta. Luego en VS Code haz git pull para que aparezca localmente.

GitHub desde Visual Studio Code

VS Code tiene integración nativa con Git. No necesitas usar la terminal — todo se puede hacer con clics — pero combinar ambos te hace más rápido y entiendes lo que pasa por debajo.

El panel de control de código (Source Control)

Se abre con Ctrl+Shift+G o pulsando el icono con forma de rama en la barra lateral izquierda. Verás:

  • Changes: archivos modificados que aún no están en staging
  • Staged Changes: archivos listos para commitear
  • Caja de mensaje: donde escribes el mensaje del commit
  • Botón ✓ Commit: equivalente a git commit

Ramas (branches): por qué son el corazón de Git

Una rama es una línea paralela de desarrollo. Te permite probar ideas, desarrollar funciones nuevas o arreglar bugs sin tocar el código que ya funciona.

Imagina esto

Tu rama principal (main) es como el manuscrito final de un libro. Cuando quieres agregar un capítulo nuevo, no escribes directamente sobre el original: sacas una copia (rama), escribes ahí, lo revisas, y solo cuando estás seguro, lo agregas al manuscrito final (merge).

Crear y trabajar con ramas

# Ver en qué rama estás y qué ramas existen
git branch

# Crear y cambiar a una rama nueva
git checkout -b feature/login

# Hacer cambios, agregar, commit...
git add .
git commit -m "implementa formulario de login"

# Subir la rama nueva al remoto
git push -u origin feature/login

# Volver a la rama principal
git checkout main

# Fusionar la rama de feature en main
git merge feature/login

# Borrar la rama ya fusionada (opcional)
git branch -d feature/login

Regla de oro

Nunca trabajes directamente en main en proyectos serios. Siempre crea una rama, haz tus cambios ahí, y luego pide que se fusione vía Pull Request. Esto protege la rama principal de errores y permite revisión.

Pull Requests: la magia colaborativa

Un Pull Request (o Merge Request en GitLab) es la pieza central de la colaboración en GitHub. Es una solicitud formal para que tu rama se fusione con otra (típicamente main), pero antes de hacerlo otras personas pueden revisar tu código, dejar comentarios, sugerir cambios y aprobar.

Crear un Pull Request paso a paso

  1. Haz tus cambios en una rama (nunca directamente en main).
  2. Sube la rama: git push -u origin feature/x
  3. Ve a GitHub. Aparecerá un banner "Compare & pull request". Pulsa.
  4. Escribe un título claro y una descripción que explique qué y por qué.
  5. Asigna reviewers (revisores) si trabajas en equipo.
  6. Pulsa Create pull request.
  7. Espera la revisión. Si piden cambios, hazlos en tu rama local y haz push: el PR se actualiza solo.
  8. Cuando aprueben, pulsa Merge pull request y luego Delete branch (la rama ya cumplió su misión).

El secreto de un buen PR

Los estudios muestran que la calidad de la revisión cae bruscamente en PRs de más de 400 líneas — los revisores dejan de leer con atención. PRs pequeños = mejor código. Cada PR debe representar un único cambio lógico.

Estrategias de branching: GitHub Flow vs Git Flow

Existen varias formas de organizar las ramas en un proyecto. Las dos más conocidas son GitHub Flow (simple) y Git Flow (complejo). Aprende la primera, conoce la segunda.

GitHub Flow (recomendado para empezar)

Una rama principal main siempre desplegable. Cualquier cambio nace en una rama feature, pasa por PR, y se fusiona a main. Simple, rápido, ideal para proyectos personales y CI/CD.

Git Flow (proyectos grandes con releases)

Múltiples ramas con roles específicos: main (producción), develop (desarrollo), feature/, release/, hotfix/. Más estructurado pero más complejo. Hoy se considera "viejo" y se prefiere GitHub Flow + trunk-based development en la mayoría de equipos modernos.

Seguridad: SSH y Personal Access Tokens

GitHub ya no acepta tu contraseña real para operaciones desde la terminal. Necesitas SSH (recomendado) o un Personal Access Token (PAT).

Opción A: Claves SSH (la más cómoda a largo plazo)

SSH usa criptografía de clave pública: tu PC genera una pareja de llaves (pública + privada). La pública se la das a GitHub, la privada nunca sale de tu PC. Cuando haces push, GitHub te reconoce automáticamente sin pedirte contraseña.

# Generar par de llaves (acepta los defaults pulsando Enter)
ssh-keygen -t rsa -b 4096 -C "tucorreo@ejemplo.com"

# Iniciar el agente SSH
eval "$(ssh-agent -s)"

# Agregar tu llave privada al agente
ssh-add ~/.ssh/id_rsa

# Copiar la llave pública (lo que pegarás en GitHub)
cat ~/.ssh/id_rsa.pub

Luego en GitHub:

  1. Foto de perfil → Settings
  2. Menú izquierdo → SSH and GPG keys
  3. New SSH key: en Title pon el nombre de tu dispositivo (ej: "Laptop personal"), en Key type deja "Authentication Key", y en el cuadro pega lo que te dio cat ~/.ssh/id_rsa.pub
  4. Pulsa Add SSH key

Opción B: Personal Access Token (PAT)

Es una "contraseña alternativa" con permisos específicos y caducidad. Útil cuando no puedes usar SSH (ej: en cuentas corporativas).

Crear un Personal Access Token (clásico)
PasoAcción
1Foto de perfil → Settings
2Menú izquierdo (abajo) → Developer settings
3Personal access tokens → Tokens (classic)
4Generate new token (classic)
5En Note: descripción (ej: "Token laptop personal")
6En Expiration: 60 o 90 días recomendado
7En Scopes: marca repo (control total de repos)
8Generate token → cópialo ahora (no podrás verlo otra vez)

Cuando hagas git push y te pida contraseña, en lugar de tu contraseña real pega el token:

Username for 'https://github.com': tu-usuario
Password for 'https://tu-usuario@github.com': pega-el-token-aqui

Errores que JAMÁS debes cometer

1. Compartir tu clave privada (id_rsa) — es como dar tu contraseña.

2. Subir tu PAT o tu archivo .env a un repo público — los bots los detectan en minutos.

3. Pegar tu API key en código JavaScript que vaya al navegador — cualquiera la ve con F12.

Solución preventiva: usa .gitignore desde el primer commit.

Tips, buenas prácticas y trucos profesionales

Mensajes de commit que cuentan una historia

Un buen mensaje de commit explica el por qué, no el qué (el qué se ve en el diff).

Compara estos commits

❌ git commit -m "cambios"

❌ git commit -m "fix"

❌ git commit -m "actualizo index.html"

✅ git commit -m "corrige validación de email en formulario de registro"

✅ git commit -m "agrega modo oscuro al panel lateral"

Convención Conventional Commits (estándar profesional)

tipo(área): descripción corta

# Tipos comunes
feat:     nueva funcionalidad
fix:      corrección de bug
docs:     cambios en documentación
style:    formato (espacios, comas) sin cambio lógico
refactor: reorganización sin cambiar comportamiento
test:     agregar/corregir tests
chore:    tareas de mantenimiento

# Ejemplos
feat(auth): agrega login con Google
fix(cart): corrige cálculo de IVA en checkout
docs(readme): añade sección de instalación

Crear un .gitignore desde el día 1

Es un archivo que le dice a Git qué archivos nunca debe trackear: contraseñas, dependencias pesadas, archivos temporales del editor.

# Variables de entorno y secretos

# Archivos del sistema

# Archivos del editor
.vscode/

*.swp

# Build / compilación
dist/
build/

Otros tips de oro

  • Commitea seguido y pequeño: mejor 10 commits de una idea cada uno que 1 commit gigante. Es más fácil revertir y entender.
  • Pull antes de push: si trabajas en equipo, siempre haz git pull antes de empezar a programar y antes de hacer push.
  • Usa git fetch en lugar de git pull cuando quieras ver qué cambió antes de fusionarlo.
  • Aprovecha git stash: cuando te llaman para arreglar algo urgente en otra rama y tienes cambios sin commitear, "guárdalos en el cajón" con git stash y recupéralos después con git stash pop.
  • Configura aliases: git config --global alias.st status te permite escribir git st en lugar de git status. Multiplica tu velocidad.
  • Lee el git log gráfico: git log --oneline --graph --all --decorate es uno de los comandos más útiles que existen.
  • README es tu carta de presentación: dedícale tiempo. Debe tener: descripción, instalación, uso, capturas. Es lo primero que ve quien visita tu repo.
  • Usa GitHub Pages: cualquier repo público puede convertirse en un sitio web gratis. Settings → Pages → Source.
  • Stars y forks: dar ⭐ a proyectos que te gusten es como bookmarking. Hacer fork es clonar un repo a tu cuenta para poder modificarlo.
  • Issues como to-do list: puedes usar issues para registrar bugs y tareas pendientes en tu propio proyecto.

Comandos útiles que casi nadie enseña

# Ver historial visual y compacto
git log --oneline --graph --all --decorate

# Buscar texto dentro del historial de commits
git log -S "función_que_buscas"

# Ver quién modificó cada línea de un archivo
git blame archivo.js

# Deshacer el último commit pero mantener los cambios
git reset --soft HEAD~1

# Renombrar el último commit (antes del push)
git commit --amend -m "nuevo mensaje"

# Ver diferencias entre dos ramas
git diff main..feature/x

# Ver commits que están en una rama pero no en otra
git log main..feature/x --oneline

Errores comunes y cómo resolverlos

1. "fatal: not a git repository"

Estás en una carpeta donde no hay git init. Solución:

git init
# O verifica que estás en la carpeta correcta:
pwd

2. "Updates were rejected because the remote contains work..."

El remoto tiene commits que tu local no tiene. Solución:

git pull origin main
# Si avisa de "histories no relacionadas":
git pull origin main --allow-unrelated-histories
# Luego:
git push

3. "Please tell me who you are"

No has configurado tu identidad. Solución:

git config --global user.name "Tu Nombre"
git config --global user.email "tucorreo@ejemplo.com"

4. "Authentication failed" al hacer push

GitHub no acepta contraseña. Genera un PAT (sección Seguridad) o configura SSH.

5. Subiste un archivo con contraseñas por error

Acción urgente

1. Considera la contraseña/key comprometida: cámbiala YA en el servicio correspondiente. Aunque borres el commit, ya quedó expuesta.

2. Para borrar el archivo del historial: git filter-branch o la herramienta BFG Repo-Cleaner.

3. Agrega el archivo a .gitignore para que no vuelva a pasar.

6. Conflictos de merge

Dos personas modificaron las mismas líneas. Git no sabe qué versión usar. Cuando abres el archivo verás algo así:

<<<<<<< HEAD
const color = "azul";
=======
const color = "rojo";
>>>>>>> feature/dark-mode

Edita el archivo dejando solo la versión correcta (quita los marcadores <<<, ===, >>>), guarda, y haz:

git add archivo-en-conflicto.js
git commit -m "resuelve conflicto de merge en color principal"

VS Code te ayuda con conflictos

Cuando abres un archivo en conflicto, VS Code muestra botones "Accept Current Change", "Accept Incoming Change", "Accept Both". Úsalos en lugar de editar manualmente.

7. Olvidaste hacer pull y ahora tienes conflictos

Si tus cambios son pequeños y no quieres lidiar con conflictos:

# Guarda tus cambios en el cajón
git stash

# Baja los últimos cambios del remoto
git pull

# Recupera tus cambios encima de los del remoto
git stash pop

# Si hay conflictos ahora, resuélvelos manualmente
