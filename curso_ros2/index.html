01¿Qué es ROS 2?

ROS 2 (Robot Operating System 2) no es un sistema operativo en el sentido clásico, sino un middleware y un conjunto de bibliotecas y herramientas de código abierto para construir aplicaciones robóticas. Resuelve los problemas que todo robot comparte —comunicación entre procesos, paso de mensajes, control de drivers, gestión de paquetes y herramientas de depuración— para que no tengas que reinventarlos en cada proyecto.

Definición

ROS 2 es un framework que conecta muchos programas pequeños e independientes (llamados nodos) que se comunican entre sí enviándose datos a través de una red. Cada nodo hace una tarea (leer un sensor, mover un motor, planificar una ruta) y juntos forman el comportamiento del robot.

La idea central es la modularidad: en lugar de un único programa gigante, tu robot es un grafo de nodos que se pueden desarrollar, probar, reiniciar y reemplazar de forma aislada. Esto facilita el trabajo en equipo, la reutilización de código y la depuración.

¿Para qué sirve?

  • Abstracción de hardware: hablar con sensores y actuadores mediante interfaces estándar.
  • Comunicación: paso de mensajes entre procesos, incluso entre máquinas distintas en red.
  • Ecosistema: miles de paquetes listos —navegación (Nav2), percepción, manipulación (MoveIt 2), simulación (Gazebo)—.
  • Herramientas: visualización (RViz2), grabación de datos (rosbag), inspección del grafo (rqt).

Analogía

Piensa en ROS 2 como el sistema nervioso del robot: los nodos son los órganos (ojos, músculos, cerebro) y los topics son los nervios que transportan la información entre ellos. ROS 2 garantiza que el mensaje llegue del sensor al cerebro y del cerebro al motor.

02ROS 1 vs ROS 2: por qué cambió todo

ROS 1 nació en 2007 para investigación. ROS 2 es una reescritura pensada para producción, robots reales, multi-robot y sistemas críticos. El cambio más importante: ROS 1 dependía de un proceso central, el roscore (el ROS Master); ROS 2 lo elimina y usa DDS con descubrimiento descentralizado.

Importante

04Conceptos clave

Estos seis conceptos son el vocabulario que usarás todo el tiempo. Entiéndelos bien y el resto del curso fluye.

🔵

Nodo (Node)

Programa que cumple una función concreta. Un robot ejecuta muchos nodos a la vez (cámara, ruedas, planificador).

📡

Topic

Canal con nombre por el que fluyen datos en streaming. Modelo publicador/suscriptor, muchos a muchos.

📨

Mensaje (Message)

La estructura de datos que viaja por un topic (p. ej. geometry_msgs/Twist para velocidad).

🤝

Servicio (Service)

Llamada petición/respuesta, síncrona. Un cliente pregunta, un servidor responde una vez.

🎯

Acción (Action)

Tarea de larga duración con feedback y posibilidad de cancelar (p. ej. "ve a este punto").

⚙️

Parámetro (Parameter)

Valor de configuración de un nodo, ajustable en tiempo de ejecución (velocidad máxima, ganancias).

¿Cuándo usar cada mecanismo de comunicación?

05Arquitectura y DDS

El robot, visto desde fuera, es un grafo de cómputo: nodos (vértices) conectados por topics, servicios y acciones (aristas). No hay un centro: cualquier nodo descubre a los demás automáticamente gracias a DDS.

flowchart LR
    CAM["/camera_node
📷 Cámara"]
    LIDAR["/lidar_node
📡 LIDAR"]
    BRAIN["/navigation
🧠 Planificador"]
    MOTOR["/base_controller
⚙️ Motores"]

    CAM -- "/image_raw" --> BRAIN
    LIDAR -- "/scan" --> BRAIN
    BRAIN -- "/cmd_vel" --> MOTOR
    MOTOR -- "/odom" --> BRAIN

Figura 1. Grafo de cómputo típico: los sensores publican, el cerebro procesa y ordena, el controlador mueve y devuelve odometría.

La pila de capas de ROS 2

Tu código nunca habla directamente con DDS. Lo hace a través de capas que te aíslan de los detalles:

    A["Tu nodo
(Python o C++)"]
    B["rclpy / rclcpp
API por lenguaje"]
    C["rcl
Biblioteca cliente común (C)"]
    D["rmw
ROS Middleware Interface"]
    E["DDS
Fast DDS · Cyclone DDS · …"]
    A --> B --> C --> D --> E

Figura 2. Capas de ROS 2. Cambiar de implementación DDS solo afecta a la capa rmw; tu código no cambia.

DDS y QoS

DDS (Data Distribution Service) es el estándar industrial de comunicación que ROS 2 usa por debajo. Aporta el descubrimiento automático (los nodos se encuentran solos) y la QoS (Quality of Service): puedes pedir entrega fiable o "best effort", definir cuántos mensajes guardar en cola, etc. Esto permite ajustar la comunicación a una red WiFi inestable o a un sensor crítico.

Cuidado con la QoS incompatible

Si un publicador usa QoS "best effort" y un suscriptor exige "reliable", no se conectarán aunque el topic sea el mismo. Es una de las causas más frecuentes de "publico pero nadie recibe".

06Instalación paso a paso

Instalaremos ROS 2 Jazzy sobre Ubuntu 24.04 con paquetes Debian (el método oficial y más estable). Si usas Humble + Ubuntu 22.04, sustituye jazzy por humble en todos los comandos.

¿No tienes Ubuntu?

Opciones: una máquina virtual (VirtualBox/VMware), WSL 2 en Windows, dual boot, o Docker con la imagen oficial ros:jazzy. Para empezar, una VM o WSL 2 es lo más sencillo.

  1. Configurar el locale (UTF-8)

    ROS 2 requiere un locale con soporte UTF-8.

    locale  # comprueba si ya tienes UTF-8
sudo apt update && sudo apt install -y locales
sudo locale-gen en_US en_US.UTF-8
sudo update-locale LC_ALL=en_US.UTF-8 LANG=en_US.UTF-8
export LANG=en_US.UTF-8

  2. Habilitar el repositorio Universe

    sudo apt install -y software-properties-common
sudo add-apt-repository universe

  3. Añadir la clave GPG y el repositorio de ROS 2

    El paquete ros2-apt-source configura la clave y la lista de fuentes, y las mantiene actualizadas automáticamente.

    sudo apt update && sudo apt install -y curl
export ROS_APT_SOURCE_VERSION=$(curl -s \
  https://api.github.com/repos/ros-infrastructure/ros-apt-source/releases/latest \
  | grep -F "tag_name" | awk -F\" '{print $4}')
curl -L -o /tmp/ros2-apt-source.deb \
  "https://github.com/ros-infrastructure/ros-apt-source/releases/download/${ROS_APT_SOURCE_VERSION}/ros2-apt-source_${ROS_APT_SOURCE_VERSION}.$(. /etc/os-release && echo $VERSION_CODENAME)_all.deb"
sudo apt install -y /tmp/ros2-apt-source.deb

  4. Actualizar e instalar ROS 2

    desktop incluye RViz2, demos y tutoriales (recomendado). ros-base es la versión mínima sin GUI, ideal para robots embebidos.

    sudo apt update && sudo apt upgrade -y
sudo apt install -y ros-jazzy-desktop      # completo (recomendado)
# sudo apt install -y ros-jazzy-ros-base  # mínimo, sin GUI

  5. Instalar herramientas de desarrollo

    Necesarias para compilar tus propios paquetes (colcon, rosdep, etc.).

    sudo apt install -y ros-dev-tools

  6. Verificar la instalación

    Carga el entorno y lanza un par de nodos de demostración (ver siguiente sección).

    source /opt/ros/jazzy/setup.bash
ros2 doctor          # diagnóstico general del sistema
ros2 --help

Error típico: "ros2: command not found"

Significa que no has cargado el entorno. Ejecuta source /opt/ros/jazzy/setup.bash en esa terminal. Para no repetirlo en cada terminal nueva, automatízalo en la siguiente sección.

07Configurar el entorno (sourcing)

ROS 2 vive en /opt/ros/<distro>. Para que tu terminal "sepa" dónde están los comandos y paquetes, debes cargar (source) su script de setup. Hacerlo a mano cada vez es tedioso, así que lo añadimos a ~/.bashrc para que se cargue solo.

# Cargar ROS 2 automáticamente en cada terminal nueva
echo "source /opt/ros/jazzy/setup.bash" >> ~/.bashrc
source ~/.bashrc

# Comprobar que las variables de ROS están definidas
printenv | grep -i ROS

Variables de entorno útiles

Tabla 4 — Variables de entorno frecuentes
VariablePara qué sirve

# Ejemplo: aislar tu sesión en una red compartida (aula, laboratorio)
echo "export ROS_DOMAIN_ID=42" >> ~/.bashrc
echo "export ROS_AUTOMATIC_DISCOVERY_RANGE=LOCALHOST" >> ~/.bashrc
source ~/.bashrc

Regla de oro del "sourcing"

Habrá dos setups que cargar: el de ROS 2 (/opt/ros/jazzy/setup.bash, el underlay) y el de tu propio espacio de trabajo (~/ros2_ws/install/setup.bash, el overlay). Siempre se carga primero el underlay y luego el overlay.

08Primera prueba: turtlesim

turtlesim es el "Hola Mundo" de ROS: un simulador de una tortuga que se mueve por la pantalla. Es perfecto para ver nodos, topics y servicios en acción sin tocar código.

sudo apt install -y ros-jazzy-turtlesim   # si no está ya instalado

Abre tres terminales (recuerda que ROS ya se carga solo desde el ~/.bashrc):

  1. Terminal 1 — Lanzar el simulador

    ros2 run turtlesim turtlesim_node

    Aparece una ventana con una tortuga.

  2. Terminal 2 — Controlarla con el teclado

    ros2 run turtlesim turtle_teleop_key

    Con esta terminal enfocada, mueve la tortuga con las flechas.

  3. Terminal 3 — Inspeccionar lo que ocurre

    ros2 node list            # nodos activos
ros2 topic list           # canales de datos
ros2 topic echo /turtle1/cmd_vel   # ve las órdenes de velocidad
rqt_graph                 # dibuja el grafo en una ventana

Mueve la tortuga sin teclado

Publica una velocidad directamente y observa cómo gira:

ros2 topic pub --rate 1 /turtle1/cmd_vel geometry_msgs/msg/Twist \
  "{linear: {x: 2.0}, angular: {z: 1.8}}"

09Workspace y colcon

Un workspace (espacio de trabajo) es una carpeta donde guardas y compilas tus propios paquetes. La convención es ~/ros2_ws con una subcarpeta src/ dentro.

flowchart TB
    WS["ros2_ws/"]
    SRC["src/
📦 tu código fuente"]
    BUILD["build/
🔧 archivos intermedios"]
    INSTALL["install/
✅ ejecutables + setup.bash"]
    LOG["log/
📝 registros de compilación"]
    WS --> SRC
    WS --> BUILD
    WS --> INSTALL
    WS --> LOG
    SRC -. "colcon build" .-> BUILD
    BUILD -. genera .-> INSTALL

Figura 3. Estructura de un workspace. Solo src/ lo escribes tú; el resto lo genera colcon.

  1. Crear la estructura

    mkdir -p ~/ros2_ws/src
cd ~/ros2_ws

  2. Resolver dependencias con rosdep

    rosdep instala automáticamente las dependencias del sistema que declaran tus paquetes.

    sudo rosdep init      # solo la primera vez
rosdep update
rosdep install --from-paths src --ignore-src -r -y

  3. Compilar con colcon

    colcon build --symlink-install

    --symlink-install enlaza los archivos en vez de copiarlos: al editar scripts de Python no necesitas recompilar.

  4. Cargar tu workspace (overlay)

    source install/setup.bash

    Hazlo en cada terminal donde quieras usar tus paquetes, o añádelo al ~/.bashrc.

Opciones de colcon que más usarás

Tabla 5 — Comandos de colcon
ComandoQué hace

colcon build --symlink-installEnlaza en vez de copiar (ideal para Python)

Ejecuta colcon build desde ~/ros2_ws (la carpeta que contiene src/), nunca desde dentro de src/. Tras compilar, abre una terminal nueva o vuelve a hacer source install/setup.bash.

10Crear un paquete

Un paquete es la unidad mínima de organización y distribución de código en ROS 2. Puede ser de Python (ament_python) o de C++ (ament_cmake).

Paquete de Python

cd ~/ros2_ws/src
ros2 pkg create --build-type ament_python --license Apache-2.0 \
  --node-name mi_nodo mi_paquete

Paquete de C++

cd ~/ros2_ws/src
ros2 pkg create --build-type ament_cmake --license Apache-2.0 \
  --node-name mi_nodo mi_paquete_cpp

¿Qué contiene un paquete?

Python o C++, ¿cuál?

Python (rclpy): más rápido de escribir, ideal para prototipos, lógica de alto nivel y aprender. C++ (rclcpp): máximo rendimiento y control fino, para bucles de control en tiempo real o nodos exigentes. ROS 2 mezcla ambos sin problema en el mismo robot.

11Nodos

Un nodo es la unidad de ejecución. En ROS 2 se crea heredando de la clase Node. El esqueleto mínimo en Python:

import rclpy
from rclpy.node import Node


class MiNodo(Node):
    def __init__(self):
        super().__init__('mi_nodo')          # nombre del nodo
        self.get_logger().info('¡Nodo iniciado!')


def main(args=None):
    rclpy.init(args=args)                    # inicializa ROS 2
    nodo = MiNodo()
    rclpy.spin(nodo)                         # mantiene el nodo vivo
    nodo.destroy_node()
    rclpy.shutdown()                         # cierra ROS 2 limpiamente


if __name__ == '__main__':
    main()

¿Qué es rclpy.spin()?

Es el bucle de eventos: mantiene el nodo "vivo" esperando mensajes, llamadas a servicios y temporizadores, y ejecuta los callbacks correspondientes. Sin spin, el nodo arrancaría y terminaría de inmediato.

Registrar el ejecutable (setup.py)

Para poder lanzarlo con ros2 run, declara el entry point en setup.py:

entry_points={
    'console_scripts': [
        'mi_nodo = mi_paquete.mi_nodo:main',
    ],
},

Compila y ejecuta:

cd ~/ros2_ws
colcon build --packages-select mi_paquete
source install/setup.bash
ros2 run mi_paquete mi_nodo

12Topics: publicador y suscriptor

Los topics son el mecanismo más usado: un nodo publica mensajes en un canal con nombre y otros nodos los reciben suscribiéndose. Es asíncrono y de muchos a muchos.

sequenceDiagram
    participant P as Publicador
    participant T as Topic /chatter
    participant S as Suscriptor
    loop cada 0.5 s
        P->>T: publish("Hola N")
        T->>S: entrega "Hola N"
        S->>S: callback procesa el mensaje
    end

Figura 4. Flujo publicador → topic → suscriptor.

Publicador (Python)

import rclpy
from rclpy.node import Node
from std_msgs.msg import String


class Publicador(Node):
    def __init__(self):
        super().__init__('publicador')
        # (tipo de mensaje, nombre del topic, tamaño de cola)
        self.pub = self.create_publisher(String, 'chatter', 10)
        self.timer = self.create_timer(0.5, self.tick)   # 2 Hz
        self.i = 0

    def tick(self):
        msg = String()
        msg.data = f'Hola ROS 2: {self.i}'
        self.pub.publish(msg)
        self.get_logger().info(f'Publicando: "{msg.data}"')
        self.i += 1


def main(args=None):
    rclpy.init(args=args)
    rclpy.spin(Publicador())
    rclpy.shutdown()


if __name__ == '__main__':
    main()

Suscriptor (Python)

import rclpy
from rclpy.node import Node
from std_msgs.msg import String


class Suscriptor(Node):
    def __init__(self):
        super().__init__('suscriptor')
        self.sub = self.create_subscription(
            String, 'chatter', self.callback, 10)

    def callback(self, msg):
        self.get_logger().info(f'Recibido: "{msg.data}"')


def main(args=None):
    rclpy.init(args=args)
    rclpy.spin(Suscriptor())
    rclpy.shutdown()


if __name__ == '__main__':
    main()

Probarlos sin escribir código de control

ros2 run mi_paquete publicador      # terminal 1
ros2 run mi_paquete suscriptor      # terminal 2

# Inspección desde una tercera terminal:
ros2 topic list
ros2 topic echo /chatter            # ver los mensajes en vivo
ros2 topic hz /chatter              # frecuencia de publicación
ros2 topic info /chatter --verbose  # tipo, QoS y conexiones

Demo lista para usar

ROS 2 trae un publicador/suscriptor de ejemplo. Pruébalo sin compilar nada:

ros2 run demo_nodes_cpp talker     # publicador
ros2 run demo_nodes_py listener     # suscriptor

13Servicios: petición y respuesta

Un servicio es una llamada síncrona: el cliente envía una petición y espera una única respuesta. Úsalo para acciones puntuales y rápidas, no para flujos continuos.

sequenceDiagram
    participant C as Cliente
    participant S as Servidor /add_two_ints
    C->>S: Request {a: 2, b: 3}
    S->>S: callback: sum = a + b
    S-->>C: Response {sum: 5}

Figura 5. Patrón petición/respuesta de un servicio.

Servidor (Python)

import rclpy
from rclpy.node import Node
from example_interfaces.srv import AddTwoInts


class ServidorSuma(Node):
    def __init__(self):
        super().__init__('servidor_suma')
        self.srv = self.create_service(
            AddTwoInts, 'add_two_ints', self.callback)

    def callback(self, request, response):
        response.sum = request.a + request.b
        self.get_logger().info(
            f'Petición: {request.a} + {request.b} = {response.sum}')
        return response


def main(args=None):
    rclpy.init(args=args)
    rclpy.spin(ServidorSuma())
    rclpy.shutdown()


if __name__ == '__main__':
    main()

Llamarlo desde la terminal

ros2 run mi_paquete servidor_suma                 # terminal 1

ros2 service list                                  # ver servicios
ros2 service type /add_two_ints                    # ver su tipo
ros2 service call /add_two_ints example_interfaces/srv/AddTwoInts \
  "{a: 2, b: 3}"                                    # llamar

No bloquees el spin

Llamar a un servicio de forma síncrona dentro de un callback puede provocar un bloqueo (deadlock). Para clientes, usa llamadas asíncronas (call_async) y resuelve la respuesta con un future.

14Acciones: tareas largas con feedback

Las acciones son para tareas que tardan (navegar a un punto, abrir una pinza despacio). Combinan tres partes: un objetivo (goal), feedback periódico del progreso y un resultado final. Además, se pueden cancelar.

sequenceDiagram
    participant C as Cliente
    participant S as Servidor de acción
    C->>S: Goal: "ve a (4, 2)"
    S-->>C: Goal aceptado
    loop mientras avanza
        S-->>C: Feedback: "distancia restante 1.2 m"
    end
    S-->>C: Resultado: "llegué" ✅

Figura 6. Una acción = objetivo + feedback continuo + resultado, con opción de cancelar.

Estructura de una acción (.action)

int32 order          # 1. Goal: lo que pides
---
int32[] sequence     # 2. Result: lo que devuelve al terminar
---
int32[] partial_sequence  # 3. Feedback: progreso mientras trabaja

Probar una acción desde la terminal

ros2 action list                                   # acciones disponibles
ros2 action info /fibonacci                         # servidor y clientes

# Enviar un objetivo y mostrar el feedback en vivo:
ros2 action send_goal /fibonacci \
  action_tutorials_interfaces/action/Fibonacci "{order: 5}" --feedback

Topic vs Servicio vs Acción, en una frase

Topic = "te aviso constantemente". Servicio = "pregunto y me respondes ya". Acción = "ponte a ello, avísame de cómo va y dime cuando termines (o lo cancelo)".

15Parámetros

Los parámetros permiten configurar un nodo sin tocar el código: velocidad máxima, nombre de un frame, ganancias de un controlador. Se pueden leer y cambiar en tiempo de ejecución.

Declarar y usar parámetros (Python)

import rclpy
from rclpy.node import Node


class NodoConfig(Node):
    def __init__(self):
        super().__init__('nodo_config')
        # declarar con un valor por defecto
        self.declare_parameter('velocidad_max', 1.5)
        v = self.get_parameter('velocidad_max').value
        self.get_logger().info(f'velocidad_max = {v}')


def main(args=None):
    rclpy.init(args=args)
    rclpy.spin(NodoConfig())
    rclpy.shutdown()


if __name__ == '__main__':
    main()

Gestionar parámetros desde la terminal

ros2 param list                              # todos los parámetros
ros2 param get /nodo_config velocidad_max    # leer uno
ros2 param set /nodo_config velocidad_max 2.0  # cambiarlo en caliente
ros2 param dump /nodo_config                 # exportar a YAML
ros2 param load /nodo_config config.yaml     # cargar desde YAML

También puedes pasar parámetros al arrancar el nodo:

ros2 run mi_paquete nodo_config --ros-args -p velocidad_max:=3.0

Archivos YAML de parámetros

Para robots reales no se pasan parámetros uno a uno: se cargan desde un archivo .yaml dentro de un launch file. Así separas configuración de código y mantienes perfiles (simulación, real, demo).

16Interfaces personalizadas (msg / srv / action)

Cuando los tipos estándar no encajan, defines tus propias interfaces: la estructura de tus mensajes (.msg), servicios (.srv) y acciones (.action). Se generan en un paquete ament_cmake.

  1. Crear un paquete de interfaces

    cd ~/ros2_ws/src
ros2 pkg create --build-type ament_cmake mis_interfaces
mkdir -p mis_interfaces/msg mis_interfaces/srv

  2. Definir un mensaje

    Archivo msg/Temperatura.msg:

    float64 celsius
string  sensor_id
int64   timestamp

  3. Declarar la generación en CMakeLists.txt

    find_package(rosidl_default_generators REQUIRED)

rosidl_generate_interfaces(${PROJECT_NAME}
  "msg/Temperatura.msg"
  "srv/Suma.srv"
)

  4. Añadir dependencias en package.xml

    <buildtool_depend>rosidl_default_generators</buildtool_depend>
<exec_depend>rosidl_default_runtime</exec_depend>
<member_of_group>rosidl_interface_packages</member_of_group>

  5. Compilar y verificar

    cd ~/ros2_ws
colcon build --packages-select mis_interfaces
source install/setup.bash
ros2 interface show mis_interfaces/msg/Temperatura

17Archivos launch

Un robot real arranca decenas de nodos. Lanzarlos a mano uno por uno sería inviable. Los launch files arrancan y configuran varios nodos a la vez con un solo comando.

Launch en Python

Crea launch/demo.launch.py dentro de tu paquete:

from launch import LaunchDescription
from launch_ros.actions import Node


def generate_launch_description():
    return LaunchDescription([
        Node(
            package='turtlesim',
            executable='turtlesim_node',
            name='sim'
        ),
        Node(
            package='mi_paquete',
            executable='publicador',
            name='publicador',
            parameters=[{'velocidad_max': 2.0}]
        ),
    ])

Ejecutarlo

ros2 launch mi_paquete demo.launch.py

Instala la carpeta launch

Para que ros2 launch encuentre el archivo, debes instalar la carpeta launch/. En paquetes Python, añade en setup.py dentro de data_files:

import os
from glob import glob
# ...
data_files=[
    # ...
    (os.path.join('share', package_name, 'launch'),
     glob(os.path.join('launch', '*launch.py'))),
],

¿Python, XML o YAML?

ROS 2 admite launch files en los tres formatos. Python es el más potente (lógica, condicionales, variables); XML y YAML son más simples y declarativos. Para empezar, usa Python.

18Herramientas imprescindibles

ROS 2 brilla por su ecosistema de herramientas de inspección, visualización y depuración. Estas son las que usarás a diario.

Dibuja el grafo de nodos y topics en vivo. Perfecto para ver "quién habla con quién".

🖼️

💾

Graba y reproduce todos los mensajes de una sesión. Indispensable para depurar y entrenar modelos.

🌳

tf2

Sistema de transformaciones: relaciona los sistemas de coordenadas (base, sensor, mundo) en el tiempo.

🩺

🐢

Simulador físico 3D. Prueba tu robot sin hardware real antes de pasar al mundo físico.

Grabar y reproducir con ros2 bag

20Buenas prácticas y errores comunes

Buenas prácticas

  • Nombra bien tus nodos y topics: usa nombres descriptivos y en minúsculas con guiones bajos (/camera_front/image_raw).
  • Un nodo, una responsabilidad: evita nodos gigantes. La modularidad facilita pruebas y reutilización.
  • Configura con parámetros, no con constantes: nada de valores "mágicos" incrustados en el código.
  • Usa launch files + YAML desde el principio para arrancar el sistema de forma reproducible.
  • Compila con --symlink-install para iterar rápido en Python.
  • Versiona tu workspace con Git, pero ignora build/, install/ y log/ en el .gitignore.

El flujo de trabajo, en 6 pasos

  1. Carga ROS 2: source /opt/ros/<distro>/setup.bash
  2. Crea/edita tu paquete dentro de ~/ros2_ws/src
  3. Resuelve dependencias: rosdep install …
  4. Compila: colcon build --symlink-install
  5. Carga tu overlay: source install/setup.bash
  6. Ejecuta y depura: ros2 run …, rqt_graph, ros2 doctor
